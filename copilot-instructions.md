# BigCalcs 项目工作规范

本文件定义团队协作的工作方法，与 DNF 伤害计算领域知识无关。
领域知识请参阅 `AGENTS.md`。

---

## 1. 截图问答规范

当用户以截图+圈出方式提问「某个功能在哪」时：
- **优先**：在原图上标注位置后返回（当前工具集的图片编辑能力有限，若无法标注，见下方替代方案）
- **替代**：提供精确的文字定位描述，包含参照物（如「XX 按钮右侧」「面板右上角紧挨 XX 图标」）
- 如果是浏览器页面，可使用浏览器工具高亮目标元素

---

## 2. 改动流程：先总结，再确认，后执行

**严禁**收到需求后直接动手修改代码。必须遵循三步：

1. **总结**：用简短文字复述你理解的需求，列出将要修改的文件和改动内容
2. **确认**：等待用户明确回复「可以」「开始」「没问题」等肯定信号
3. **执行**：收到确认后再进行实际修改
4. **反馈**：改动完成后，简要展示修改了哪些文件、关键差异在哪里（不用贴完整 diff，突出重点变更即可）

> 例外：纯信息查询（如「XX 文件在哪里」）不需要此流程。

---

## 3. Git 推送控制

**禁止自动推送 GitHub。** 代码修改完成后只在本地提交（`git commit`），**不**主动执行 `git push`。

推送由用户发号施令——只有收到明确的「推送」「push」「上传到 GitHub」等指令时才执行。

---

## 4. 工作区命令免确认

在本工作区（BigCalcs）内部，**常规终端命令默认同意执行**，无需等待用户确认，且**不在对话中展示命令和输出**（静默执行）。包括但不限于：

- `git add` / `git commit`（本地提交）
- `mkdir` / `mv` 等文件整理
- 安装项目依赖（如 `npm install`）

> 例外：`git push` 仍须遵守第 3 条，等待明确指令。
> 例外：涉及工作区外路径或可能影响系统全局的操作，仍需先总结确认。
> 例外：命令执行出错时，才在对话中报告错误信息。

---

## 5. 变量命名规范

### 基础规则
- 使用 **小驼峰命名**（camelCase）：`skillDamage`、`baseAttack`
- 层级关系用 **下划线 `_`** 分隔：`damage_80`、`damage_80_2`、`effect_cooldown`
  - 注意：`-` 在 JavaScript 中是减号运算符，不可用于变量名

### 变量名手册
项目维护一份变量名索引，格式为两列：

| 变量名 | 描述 |
|--------|------|
| `skillDamage` | 技能伤害总值 |
| `damage_80` | 80 级技能伤害分支 |
| `damage_80_2` | 80 级技能第二部分分支 |

> 每次新增变量时，同步更新手册。手册文件建议放在 `docs/variables.md`。

---

## 6. 对话风格

- **专业、严谨、简洁**，不花哨
- 遇到描述歧义时，**主动提问澄清**，不要猜测后直接执行
- 中文交流，术语可用英文（如 `damage zone`、`modifier`）

---

## 7. 三次纠错触发重新审视

如果同一问题被用户纠正 **3 次**（每次用户表达的是相同意图，但执行结果始终不对），必须：

1. **停止当前方向**
2. **回滚前 2 次错误改动**——恢复到问题首次提出前的代码状态，不要在错误基础上继续修改
3. **重新分析问题本身**——极可能是自然语言描述与编码理解之间发生了偏差
4. **主动向用户提问**，用你自己的话复述理解，并请求用户确认或补充示例
5. **优先用 Mermaid 图表画草图**来对齐对布局/结构/流程的理解，避免纯文字描述再次产生歧义

> 典型错误模式：用户要「并排排列」，实际做出来却是「嵌套排列」——这说明对布局关系的理解完全错了，必须停下来重新对齐。
